## 2:00–5:30 DB 구조와 저장 이유

### 2:00–2:40 관계도

GitHub의 `docs/database.md` 관계도를 보여준다.

“총 7개 테이블입니다. `users`는 계정, `sessions`는 로그인 상태, `restaurants`와 `menu_items`는 식당과 메뉴, `orders`와 `order_items`는 주문 헤더와 여러 메뉴, `order_status_history`는 상태 변경 이력입니다.”

### 2:40–3:35 왜 주문을 둘로 나눴는가

“한 주문에는 메뉴가 여러 개라서 배송지와 총액을 메뉴마다 반복하면 수정 오류와 중복이 생깁니다. 그래서 주문 공통값은 `orders`, 반복되는 메뉴는 `order_items`로 나눴습니다. 특히 메뉴 FK만 저장하지 않고 주문 당시 이름과 단가를 복사합니다. 식당이 내일 가격을 바꾸거나 메뉴를 삭제해도 오늘 결제한 기록은 바뀌면 안 되기 때문입니다.”

### 3:35–4:15 상태·세션 분리 이유

“현재 상태는 주문 목록에서 자주 읽으므로 `orders.status`에 둡니다. 변경 과정은 `order_status_history`에 계속 추가해 누가 언제 바꿨는지 남깁니다. 계정과 세션도 분리했습니다. 한 사용자가 여러 기기에서 로그인할 수 있고 세션마다 만료 시각이 다르며, DB에는 탈취 위험을 줄이기 위해 토큰 원문 대신 SHA-256 해시를 저장합니다.”

### 4:15–5:05 주문 저장 흐름

“주문 요청이 오면 먼저 로그인 사용자를 확인합니다. 식당과 메뉴를 DB에서 다시 읽어 가격·품절·최소 주문 금액을 검증하고 합계를 서버에서 재계산합니다. 그다음 하나의 트랜잭션에서 `orders` 한 행, `order_items` 여러 행, 최초 상태 이력을 저장합니다. 어느 한 단계라도 실패하면 전체 롤백되어 메뉴 없는 반쪽 주문이 남지 않습니다.”

### 5:05–5:30 중복·권한 방어

“주문 폼의 UUID 요청 키에는 고유 제약이 있어 네트워크 지연 중 버튼을 다시 눌러도 한 건만 저장됩니다. 주문 조회 조건에는 현재 사용자 ID를 함께 넣어 다른 사람의 주문 ID를 알아도 내용을 볼 수 없습니다.”

## 5:30–8:25 실제 버그 — CSS 캐스케이드 레이어

### 5:30–6:00 증상

`docs/bug-log.md`의 BUG-004를 보여준다.

“가장 기억에 남은 실제 버그는 검은 버튼에서 글자가 사라진 문제입니다. 콘솔 오류는 없었지만 회원가입과 식당 둘러보기 링크가 검은 알약 모양으로만 보였습니다.”

가능하면 DevTools Styles에서 임시로 아래 규칙을 추가해 고장 화면을 재현한다.

```css
a { color: inherit; }
```

### 6:00–6:45 잘못된 첫 추측과 측정

“처음에는 Tailwind가 `text-white`를 생성하지 않았거나 캡처 문제라고 생각했습니다. 하지만 Elements에서 버튼을 선택하고 계산된 스타일을 확인하니 배경과 글자색이 둘 다 `rgb(33, 31, 28)`이었습니다. 즉 클래스가 없던 것이 아니라 다른 CSS가 덮고 있었습니다.”

개발자 도구 Console에서 선택한 링크에 `getComputedStyle($0).color`를 실행해 값을 보여준다.

### 6:45–7:35 실제 원인

`src/app/globals.css`와 커밋 `e62c1e3`의 diff를 보여준다.

“원인은 전역 `a { color: inherit }`였습니다. 단순히 선택자 명시도만 보면 클래스인 `text-white`가 강해 보이지만, 전역 규칙은 CSS 레이어 밖에 있었고 Tailwind 유틸리티는 `@layer utilities` 안에 있었습니다. CSS 캐스케이드에서는 레이어 밖 작성자 규칙이 레이어 안 규칙보다 우선해 전역 링크 색이 이겼습니다.”

### 7:35–8:05 해결과 검증

“Tailwind의 기본 초기화가 링크 스타일을 이미 처리하므로 이 전역 규칙을 삭제했습니다. `!important`로 덮으면 같은 문제가 다른 링크에서 반복될 수 있어 원인 규칙 자체를 제거했습니다.”

DevTools에서 임시 규칙을 끄고 글자가 다시 흰색이 되는 것과 `getComputedStyle($0).color`가 `rgb(255, 255, 255)`인 것을 보여준다.

### 8:05–8:25 배운 점

“이 문제로 CSS는 명시도뿐 아니라 origin, importance, cascade layer, source order를 함께 봐야 한다는 점을 배웠습니다. 이후에는 눈으로만 보지 않고 계산된 스타일로 원인을 검증했습니다.”

## 8:25–9:35 추가 기능과 검증

“추가 기능으로 식당 이름·카테고리·배달비·최소 주문 필터, 관리자 주문 상태 변경, 고객 상태 타임라인, 품절 처리, 모바일 하단 내비게이션을 구현했습니다.”

“자동 검증은 가격 계산과 인증·권한 등 Vitest 28개, TypeScript와 ESLint, 프로덕션 빌드, 실제 Chrome Playwright로 구성했습니다. Playwright는 회원가입, 일반 사용자 관리자 차단, 주문 생성, 관리자 조리 중 변경, 고객 재확인, 검색 필터, 360px 가로 넘침까지 검사하고 테스트 데이터는 자동 삭제합니다.”

## 9:35–10:00 한계와 마무리

“한계도 있습니다. 실제 결제, 이메일 인증, 실시간 라이더 위치는 외부 서비스와 보안·비용 범위 때문에 구현하지 않고 주문 저장과 상태 타임라인으로 제한했습니다. AI가 제안한 코드는 그대로 믿지 않고 실제 브라우저와 DB, 자동 테스트로 다시 검증했습니다. 프로젝트를 통해 화면 기능보다 데이터가 왜 나뉘고 실패할 때 어떻게 일관성을 지키는지가 더 중요하다는 점을 배웠습니다. 감사합니다.”
