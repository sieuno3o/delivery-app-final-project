### 0:15~0:50 회원가입 → 로그아웃 → 로그인

**동작:** `회원가입` → 이름·새 이메일·비밀번호 입력 → 가입 완료 → `로그아웃` → 방금 만든 계정으로 `로그인`

> 주문이 정상 접수됐습니다. 주문번호, 주문 당시 메뉴와 가격, 배송지, 총액, 현재 상태가 보입니다. 주문내역으로 이동해 방금 저장된 주문을 데이터베이스에서 다시 불러오겠습니다.

> 방금 사용한 식당 검색·카테고리 필터와 주문 상태 타임라인은 필수 흐름 위에 구현한 추가 기능입니다.

## ② 2:20~5:40 DB 구조와 설계 이유

### 2:20~2:55 테이블 7개와 역할

**화면:** GitHub `docs/database.md`의 상세 ERD를 한 화면에 보인 뒤 PK/FK 표까지 천천히 내린다.

> 데이터베이스는 총 7개 테이블로 구성했습니다. `users`는 회원 계정, `sessions`는 로그인 상태, `restaurants`와 `menu_items`는 식당과 메뉴를 저장합니다. `orders`는 주문 전체 정보, `order_items`는 주문에 포함된 여러 메뉴, `order_status_history`는 상태 변경 이력을 저장합니다.

### 2:55~3:35 PK/FK 관계

**화면:** 관계도에서 연결선을 차례로 가리킨다.

**주의:** 표의 모든 셀을 읽지 말고 `orders`, `order_items`, `order_status_history`의 PK/FK 행만 짚는다.

> 모든 테이블의 기본키, 즉 PK는 `id`입니다. 먼저 `orders.user_id`가 `users.id`를 참조해 주문자를 연결합니다. `orders.restaurant_id`와 `menu_items.restaurant_id`는 `restaurants.id`를 참조합니다. `order_items.order_id`는 `orders.id`, `menu_item_id`는 `menu_items.id`를 참조해 한 주문의 메뉴들을 연결합니다. 마지막으로 `order_status_history.order_id`는 주문을, `changed_by_user_id`는 상태를 바꾼 사용자를 연결합니다. 이 PK와 FK 덕분에 사용자와 식당 정보를 주문마다 중복 저장하지 않고 관계를 따라 조회할 수 있습니다.

### 3:35~4:25 주문과 주문 상세를 나눈 이유

> 주문 한 건에는 메뉴가 여러 개 들어갈 수 있습니다. 배송지와 총액을 메뉴마다 반복 저장하면 중복이 생기고, 한 곳만 수정되어 데이터가 서로 달라질 수 있습니다. 그래서 배송지, 주문자, 총액처럼 주문 전체에 한 번만 필요한 값은 `orders`에 두고, 메뉴명, 단가, 수량처럼 반복되는 값은 `order_items`로 분리했습니다.

> 또 `order_items`에는 메뉴 외래키만 저장하지 않고 주문 당시의 메뉴명과 단가를 복사해 둡니다. 식당이 나중에 메뉴 이름이나 가격을 바꾸거나 메뉴를 삭제해도 이미 주문한 기록은 바뀌면 안 되기 때문입니다.

### 4:25~5:10 주문 데이터가 저장되는 순서

**화면:** `주문 저장 순서` 항목을 가리킨다.

> 주문 요청이 오면 먼저 로그인 사용자를 확인합니다. 그다음 식당과 메뉴를 DB에서 다시 읽어 현재 가격, 품절 여부, 최소 주문 금액을 검증하고 서버에서 합계를 다시 계산합니다.

> 검증이 끝나면 하나의 트랜잭션 안에서 `orders` 한 행, `order_items` 여러 행, `order_status_history`의 최초 접수 이력을 저장합니다. 세 저장 중 하나라도 실패하면 모두 롤백합니다. 이렇게 해야 주문 정보만 있고 주문 메뉴는 없는 반쪽짜리 데이터가 남지 않습니다. 또한 UUID 요청 키에 고유 제약을 두어 네트워크 지연 중 버튼이 다시 눌려도 같은 주문이 두 번 저장되지 않게 했습니다.

> 주문 상세와 내역을 조회할 때는 주문 ID만 사용하지 않고 현재 로그인한 `user_id`도 함께 조건에 넣어 다른 사용자의 주문을 볼 수 없게 했습니다.

### 5:10~5:40 세션·상태 이력과 실행 환경

> 사용자와 세션을 나눈 이유는 한 사용자가 여러 기기에서 로그인할 수 있고 세션마다 만료 시각이 다르기 때문입니다. DB에는 세션 토큰 원문 대신 SHA-256 해시를 저장합니다. 주문의 현재 상태는 빠른 목록 조회를 위해 `orders.status`에 두고, 변경 과정은 이력 테이블에 계속 추가합니다.

> 로컬에서는 Docker Compose로 PostgreSQL 환경을 맞췄고, 배포 환경에서는 Vercel과 Neon PostgreSQL을 사용했습니다. 두 환경에 같은 Drizzle 마이그레이션을 적용해 DB 구조가 달라지는 문제를 줄였습니다.

## ③ 5:40~8:40 실제 버그 — 검은 링크에서 글자가 사라진 문제

### 5:40~6:15 증상과 재현

**화면:** 정상 홈 화면 → Mac 트랙패드 두 손가락 클릭 → `검사` → Elements에서 `<a>` 선택 → Styles의 `+` → `a { color: inherit; }` 적용

> 가장 원인을 찾기 어려웠던 버그는 검은 링크 버튼에서 흰색 글자가 사라진 문제였습니다. 단순 장식이 아니라 회원가입과 식당 이동이라는 핵심 행동을 사용자가 알아볼 수 없는 문제였습니다. 콘솔 오류 메시지는 없었기 때문에 에러 대신 화면의 실제 계산값을 증거로 확인해야 했습니다.

> 현재 코드는 수정된 상태라서, DevTools에 당시의 전역 링크 규칙을 잠시 적용해 같은 증상을 재현했습니다.

### 6:15~6:55 첫 추측과 측정

**동작:** 문제가 생긴 링크를 Elements에서 선택 → Computed 또는 Console에서 `getComputedStyle($0).color` 확인

> 처음에는 Tailwind가 `text-white` 클래스를 만들지 않았거나 화면 캡처 문제라고 추측했습니다. 하지만 Elements에서 버튼의 `<a>` 요소를 선택하고 `getComputedStyle($0).color`를 실행하니 글자색이 `rgb(33, 31, 28)`이었습니다. 배경색도 같은 값이므로 흰색 클래스가 없던 것이 아니라 다른 CSS 규칙이 글자색을 덮고 있다는 객관적인 증거를 얻었습니다.

### 6:55~7:45 실제 원인

**화면:** `docs/bug-log.md`의 BUG-004 → 커밋 `e62c1e3`의 `globals.css` diff

> 실제 원인은 `globals.css`에 있던 전역 `a { color: inherit; }` 규칙이었습니다. 처음에는 클래스 선택자인 `text-white`가 태그 선택자보다 강하다고 생각했습니다. 하지만 전역 `a` 규칙은 CSS 캐스케이드 레이어 밖에 있었고, Tailwind 유틸리티는 `@layer utilities` 안에 있었습니다.

> CSS에서는 선택자 명시도만 비교하는 것이 아니라 레이어 우선순위를 먼저 비교합니다. 그래서 레이어 밖의 전역 규칙이 레이어 안의 `text-white`보다 우선해, 부모의 검은 글자색을 링크가 상속받은 것이 정확한 원인이었습니다.

### 7:45~8:20 해결과 검증

**화면:** 커밋 diff에서 전역 `a` 규칙 삭제 확인 → DevTools 임시 규칙 해제 → 버튼 글자 복구 → 계산된 색상 재확인

> Tailwind의 기본 초기화가 링크 스타일을 이미 처리하고 있어서 불필요한 전역 `a` 규칙을 삭제했습니다. `!important`로 임시로 덮는 대신 충돌 원인 자체를 제거했습니다.

> 수정 후 같은 버튼의 계산된 글자색이 `rgb(255, 255, 255)`로 바뀌었고, 회원가입과 식당 둘러보기 글자가 모두 정상적으로 보이는 것을 확인했습니다. 실제 수정 내용은 이 커밋에도 남아 있습니다.

### 8:20~8:40 배운 점

> 이 문제를 통해 CSS 충돌은 명시도만 볼 것이 아니라 origin, importance, cascade layer, source order를 함께 확인해야 한다는 점을 배웠습니다. 이후에는 눈으로 추측하지 않고 DevTools의 계산된 스타일과 Git diff로 원인을 검증했습니다.

## ④ 8:40~9:40 회고 — AI의 한계와 남은 범위

**화면:** 완성된 서비스 홈 또는 README의 검증 항목

> AI는 코드 초안을 빠르게 만드는 데 도움이 됐지만, 배포 환경에서만 나타난 한국 시간 오류나 CSS 캐스케이드 레이어 충돌처럼 실행 환경과 브라우저 계산값이 필요한 문제는 질문만으로 바로 해결하지 못했습니다. 제안된 코드를 그대로 믿지 않고 실제 브라우저, 데이터베이스, Git diff로 원인을 다시 확인해야 했습니다.

> 실제 결제, 이메일 인증, 실시간 라이더 위치는 외부 결제사 연동, 개인정보 보호, 비용 문제 때문에 이번 범위에서는 구현하지 못했습니다. 대신 주문이 DB에 안전하게 저장되고 상태 변경 이력이 남는 핵심 흐름까지 완성했습니다.
