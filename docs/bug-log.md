# 실제 버그 기록

개발 중 직접 겪은 오류만 기록한다. 최종 문서에는 아래 기록 중 원인과 해결 과정을 가장 명확히 설명할 수 있는 3개를 선정한다.

## BUG-001 — pnpm이 Next.js 의존성 빌드 스크립트를 차단함

- 발생일: 2026-06-20
- 작업: `create-next-app`으로 프로젝트 초기화
- 에러 메시지: `ERR_PNPM_IGNORED_BUILDS: Ignored build scripts: sharp, unrs-resolver`
- 증상: 패키지는 내려받았지만 설치가 중단되어 Next.js 프로젝트 생성이 끝나지 않았다.
- 처음 확인한 내용: 네트워크 오류나 Node.js 버전 문제인지 패키지 설치 로그를 확인했다.
- 원인: pnpm 11이 의존성의 설치 스크립트를 기본적으로 실행하지 않고, 프로젝트에서 허용 여부를 명시하도록 요구했다. Next.js가 이미지 처리를 위해 사용하는 `sharp`도 차단 대상이었다.
- 해결: `pnpm-workspace.yaml`의 `allowBuilds`에 `sharp`와 `unrs-resolver`를 명시적으로 허용하고 `pnpm install`을 다시 실행했다.
- 검증: 의존성 설치가 종료 코드 0으로 완료되었고, `pnpm verify`에서 lint, 타입 검사, 테스트 2개, 프로덕션 빌드가 모두 통과했다.
- 발표 후보: 보류

## BUG-002 — PostgreSQL 컨테이너가 포트 충돌로 시작되지 않음

- 발생일: 2026-06-20
- 작업: 배달앱 로컬 PostgreSQL 컨테이너 실행
- 에러 메시지: `Bind for 0.0.0.0:5432 failed: port is already allocated`
- 증상: 컨테이너와 볼륨은 생성됐지만 DB 서비스가 시작되지 않았다.
- 처음 추측한 원인: 이전에 실패한 동일 프로젝트 컨테이너가 남아 있다고 생각했다.
- 실제 원인: `docker ps`와 `lsof`로 확인한 결과, 별도의 할 일 앱 프로젝트 `todolist_db`가 호스트의 5432 포트를 이미 사용 중이었다.
- 해결: 다른 프로젝트를 중단하지 않고 이 프로젝트의 호스트 포트를 5433으로 변경했다. 컨테이너 내부의 PostgreSQL 기본 포트 5432는 그대로 유지했다.
- 검증: 5433 포트에서 컨테이너가 `healthy` 상태가 되었고, `pnpm db:check`가 식당 6개와 메뉴 30개를 확인했다.
- 발표 후보: 예

## BUG-003 — 시드 스크립트의 top-level await 변환 실패

- 발생일: 2026-06-20
- 작업: 마이그레이션 후 식당·메뉴 시드 실행
- 에러 메시지: `Top-level await is currently not supported with the "cjs" output format`
- 증상: DB 마이그레이션은 성공했지만 `tsx scripts/seed.ts`가 실행 전에 변환 단계에서 중단됐다.
- 처음 추측한 원인: PostgreSQL 연결이나 트랜잭션 코드의 문제일 수 있다고 생각했다.
- 실제 원인: 현재 패키지가 CommonJS 형식으로 해석되는 상황에서 파일 최상위의 `await`를 esbuild가 변환할 수 없었다. DB 연결 전 발생한 모듈 형식 문제였다.
- 해결: 시드와 DB 확인 코드를 각각 `async` 함수 안으로 옮기고, 함수 호출의 실패를 `catch`에서 처리하도록 실행 진입점을 명시했다.
- 검증: 수정 후 동일한 시드 명령이 성공했고, 두 번 연속 실행해도 식당 6개와 메뉴 30개만 유지되어 중복이 생기지 않았다.
- 발표 후보: 예

## BUG-004 — 검은 링크 버튼의 글자가 보이지 않음

- 발생일: 2026-06-20
- 작업: 식당 목록 화면의 브라우저 시각 점검
- 에러 메시지: 콘솔 오류는 없었지만 `getComputedStyle`에서 버튼 배경과 글자색이 모두 `rgb(33, 31, 28)`로 확인됨
- 증상: 회원가입과 식당 둘러보기 링크가 검은 알약 모양만 보이고 흰색 글자가 사라졌다.
- 처음 추측한 원인: 브라우저 캡처 과정에서 흰색 텍스트가 누락되었거나 Tailwind가 `text-white` 클래스를 생성하지 않았다고 생각했다.
- 실제 원인: `globals.css`의 계층화되지 않은 `a { color: inherit }` 규칙이 Tailwind CSS의 `@layer utilities` 안에 있는 `text-white`보다 캐스케이드 우선순위가 높았다. 선택자 명시도만 비교하면 클래스가 높지만, CSS 레이어 밖의 규칙이 레이어 안 규칙보다 우선한다.
- 해결: 링크 스타일은 Tailwind의 기본 초기화가 이미 처리하므로, 레이어 밖에 있던 전역 `a` 규칙 전체를 제거해 컴포넌트의 `text-white`가 우선 적용되게 했다.
- 검증: 브라우저에서 두 링크의 계산된 글자색이 `rgb(255, 255, 255)`로 바뀐 것을 확인했고, 재캡처한 화면에서 글자가 정상적으로 보였다.
- 발표 후보: 예

## BUG-005 — 주문 완료 안내가 즉시 사라짐

- 발생일: 2026-06-20
- 작업: 체크아웃부터 주문 상세까지 실제 브라우저 주문 흐름 점검
- 에러 메시지: 콘솔 오류는 없었고 주문 상세 페이지가 연속으로 다시 요청됨
- 증상: 주문 저장과 장바구니 초기화는 성공했지만 `주문이 접수됐어요!` 안내가 화면에 거의 보이지 않고 일반 주문 상세 제목으로 바로 바뀌었다.
- 처음 추측한 원인: 서버 액션의 리다이렉트가 완료 안내용 `placed=1` 쿼리를 전달하지 못했다고 생각했다.
- 실제 원인: 완료 화면의 클라이언트 효과가 장바구니를 비운 직후 `router.replace`를 실행해 서버 컴포넌트를 다시 렌더링했고, 완료 안내 조건인 `placed=1`을 즉시 제거했다.
- 해결: Next.js 라우터로 다시 렌더링하지 않고 브라우저의 `history.replaceState`로 주소만 정리했다. 현재 화면의 완료 안내는 유지되고, 새로고침할 때는 일반 주문 상세가 열린다.
- 검증: 실제 주문 후 완료 안내, 주문번호, 메뉴 스냅샷, 총액이 함께 표시되고 주소창에서는 `placed=1`만 제거되는 것을 확인했다.
- 발표 후보: 예

## BUG-006 — 운영 주문 시간이 한국 시간보다 9시간 느림

- 발생일: 2026-06-20
- 작업: Vercel 공개 URL에서 회원가입부터 주문내역까지 운영 smoke test
- 에러 메시지: 콘솔 오류는 없었고 같은 주문이 로컬에서는 15시, 운영에서는 06시로 표시됨
- 증상: 주문과 생성 시각은 DB에 정상 저장됐지만 주문 상세와 주문내역의 표시 시간이 한국 시간보다 정확히 9시간 느렸다.
- 처음 추측한 원인: Neon에서 타임스탬프를 UTC로 저장한 것이 문제라고 생각했다.
- 실제 원인: DB의 `timestamptz` UTC 저장은 정상이고, `Intl.DateTimeFormat`에 시간대를 지정하지 않아 로컬에서는 macOS의 한국 시간대, Vercel에서는 서버의 UTC 시간대를 각각 사용했다.
- 해결: 두 주문 화면의 날짜 포맷에 `timeZone: "Asia/Seoul"`을 명시해 실행 환경과 관계없이 한국 시간으로 표시되게 했다.
- 검증: 수정 배포 후 기존 운영 주문 상세와 주문내역 모두 생성 시각이 15시로 표시되는지 다시 확인한다.
- 발표 후보: 예
